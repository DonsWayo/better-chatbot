## (A) Top 15 UX/Polish Improvements — Ranked

Inbox + Runs front-loaded (the user's named complaint), then by severity × cross-surface frequency.

**1. Approval decision has no in-flight or success feedback**
- Surface: Inbox / Runs
- Severity: P2 (but #1 — it's the core triage loop)
- Why it matters: Approve/reject is the primary action of the named surface. Today the page does a full `router.refresh()` with no pending state and no success confirmation — users don't know if their decision registered and may double-click.
- Fix: Disable buttons during `isPending` (already tracked via `useTransition`), show `Loader2` in the clicked button, fade out the approval panel with a brief success icon/toast before refresh. Apply in both `approval-decision-buttons.tsx` and the run-detail banner (`runs/[id]/page.tsx:150-168`).

**2. Reject flow has no confirmation, char count, or submission feedback**
- Surface: Inbox / Runs
- Severity: P2
- Why it matters: Reject is destructive and reason-required, yet it's a two-click flow with no `maxLength`, no counter, no confirmation, and only an optimistic hide. Easy to mis-fire in a triage-heavy workflow.
- Fix: `maxLength={500}` + char counter on the textarea (`approval-decision-buttons.tsx:70-77`); success toast quoting the reason excerpt before refresh. Optional AlertDialog confirm before the server action.

**3. Reusable empty-state + skeleton components across inbox/runs**
- Surface: Inbox / Runs (also Documents, Sidebar, Admin, Agents, Knowledge — same gap everywhere)
- Severity: P2
- Why it matters: All five inbox/runs empty states share identical undifferentiated dashed-box styling with no CTA or context; there are zero loading skeletons despite `Promise.all` server fetches and 5s polling. This is the single most repeated finding in the whole audit.
- Fix: Build one `<EmptyState icon heading description cta>` and one skeleton-row pattern (shadcn Card + Skeleton). Differentiate CTAs (emptyApprovals→dashboard, emptyRoutines→/schedule, emptyRuns→/chat). Reuse across inbox, sidebar runs rail, schedule dialog, documents, admin tables, agents, knowledge.

**4. Sidebar runs rail vanishes when collapsed — live runs become invisible**
- Surface: Runs (sidebar)
- Severity: P2
- Why it matters: `group-data-[collapsible=icon]:hidden` fully hides active runs when the sidebar is collapsed; power users running many jobs lose all visibility, and cost/time info is clipped at narrow widths.
- Fix: Compact collapsed view — status dot + run-count badge, hover tooltip with status label; revert to full layout when expanded (`app-sidebar-runs.tsx:143-207`).

**5. Run-detail page not responsive on mobile/tablet**
- Surface: Runs
- Severity: P2
- Why it matters: `grid-cols-2 sm:grid-cols-3` breaks on phones (timestamps/cost stack awkwardly), badges overflow without wrapping, and JSON `<details>` blocks consume the whole viewport with no copy affordance.
- Fix: `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` on metadata; `flex-wrap` on badges; enhance JsonBlock with chevron, line count, and a Copy button using the existing `CodeBlock` component (`runs/[id]/page.tsx`).

**6. Replace all `window.prompt` / `window.alert` with styled dialogs**
- Surface: Documents (link insert), Sidebar (folder rename), Agents (referenced)
- Severity: P2
- Why it matters: Raw browser prompts are unstyled, untranslatable, un-validatable, and block the page — a v1 tell that appears in at least three surfaces.
- Fix: shadcn Popover+Input for the document link dialog (`document-editor.tsx:144-154`) with basic URL validation; shadcn Dialog for folder rename (`sidebar-folders.tsx:148`), matching the existing thread-rename pattern.

**7. Message action buttons are below minimum touch target and hidden on mobile**
- Surface: Chat composer
- Severity: P1
- Why it matters: `size-3! p-4!` yields a ~6–12px tap area (WCAG violates 44px), and `md:opacity-0 group-hover/message:opacity-100` hides Copy/Edit/Delete/Feedback/Change-Model entirely on touch devices — mobile users cannot access core message actions at all.
- Fix: Bump icon buttons to `h-8 w-8` minimum; on `<md`, render a persistent `⋯` overflow menu (Copy/Edit/Delete/Feedback) instead of hover-only buttons (`message-parts.tsx:219, 226, 242, 258, 392-440`).

**8. Save-error states offer no recovery action**
- Surface: Documents (autosave), Admin (policy/budget cards), Chat, Inbox
- Severity: P2
- Why it matters: Failed saves surface only a transient toast or small red text with no Retry and no error classification (permission vs network vs server). Repeated across documents, admin memory/local-MCP cards, and approval/schedule flows.
- Fix: Replace status text with a clickable Alert exposing the parsed error + a Retry button that re-triggers the mutation (`document-editor-page.tsx:128-130`, `memory-policy-card.tsx:137`, `local-mcp-policy-card.tsx:124`).

**9. Read-only state is visually indistinguishable from editable**
- Surface: Documents, Agents
- Severity: P2
- Why it matters: Disabled title inputs still look active; toolbars simply vanish for viewers; agent forms apply `readOnly` with no visual cue. Users think the app is broken or try to edit and silently fail.
- Fix: Add a "View only" badge + lock icon and `bg-muted/30` on read-only inputs; render a disabled toolbar frame or persistent "You have view-only access" banner (`document-editor-page.tsx:419-427`, `document-editor.tsx:156-267`, `edit-agent.tsx`).

**10. Destructive actions lack confirmation + destructive styling**
- Surface: Sidebar (delete-all chats), Admin (rate-limit reset, deletes), Inbox (reject)
- Severity: P1 (sidebar delete-all) / P2
- Why it matters: "Delete all chats" and "Reset rate limit" fire with weak or no confirmation; delete dialogs use neutral (not red) buttons with no warning icon — high blast-radius mistakes are one tap away.
- Fix: Standardize on shadcn AlertDialog with `bg-destructive` button + AlertTriangle icon and an explicit count ("This will delete X chats forever"). Add confirmation to rate-limit reset (`user-admin-actions.tsx:29-41`) and sidebar delete-all (`app-sidebar-threads.tsx:150-172`).

**11. Admin team-detail page is a 4000px single scroll column**
- Surface: Admin
- Severity: P1
- Why it matters: 11+ stacked Card sections with no navigation/collapse cause severe scrolling fatigue, especially on mobile; success/error messages flicker away on `router.refresh()` so admins miss whether saves landed.
- Fix: shadcn Tabs (Overview / Members / Security / Budget) with a sticky header; replace toast-only save feedback with a persistent "Saved ✓" badge next to each save button for 2–3s (`team-detail-client.tsx`).

**12. Knowledge mentions never render in the chat @-picker**
- Surface: Knowledge / Chat
- Severity: P1
- Why it matters: `knowledgeMentions` is fetched and curated but omitted from `groupedMentions` — a partly-shipped backend feature is completely undiscoverable in the UI.
- Fix: Add a `knowledge` group to `groupedMentions` and render a Knowledge section in both mobile and desktop mention layouts (`chat-mention-input.tsx:323-355, 529-547, 663-862`).

**13. Dead skeleton code + over-broad edit permission in agent editor**
- Surface: Agents
- Severity: P1
- Why it matters: `{false ? <Skeleton/> : <Input/>}` placeholders never render (dead code), and `hasEditAccess = isOwner || agent.visibility === 'public'` wrongly grants edit to any viewer of a shared/team/company agent under the four-level model.
- Fix: Remove the dead ternaries (or wire to real `isLoading`); change to `const hasEditAccess = isOwner` — non-owner access is enforced server-side (`agent/[id]/page.tsx:38`, `edit-agent.tsx:445+`).

**14. Tool/dropdown menus overflow viewport and lack consistent spacing on mobile**
- Surface: Chat, Sidebar, Admin
- Severity: P2
- Why it matters: Tool-select dropdown has no max-height/responsive width and bleeds off-screen; admin mobile nav forces horizontal scroll through 13 items; command palette and dropdown menu items use inconsistent padding/heights.
- Fix: `w-screen sm:w-72 max-h-[70vh] overflow-y-auto` (or Sheet on `<md`) for tool dropdown (`tool-select-dropdown.tsx:248`); replace admin mobile tab bar with a Sheet drawer (`admin-sidebar.tsx:189-211`); standardize menu item `min-h-9`.

**15. Focus-visible + ARIA gaps on custom-role controls**
- Surface: Sidebar, Admin, Auth, Documents
- Severity: P1 (sidebar mobile toggle) / P2
- Why it matters: `role='button'` divs (sidebar header toggle) and AUP checkbox have no visible focus ring; audit-log filter labels lack `htmlFor`; password checklist relies on color alone (colorblind-inaccessible); step indicators lack `aria-live`. Keyboard and screen-reader users are blocked at multiple entry points.
- Fix: Replace `role='button'` divs with `Button` or add `focus-visible:ring-2` (`sidebar-header.tsx:78-98`); wire `htmlFor`/`id` on audit-log filters; add text/`aria-label` to password rules; `aria-live='polite'` on the signup step indicator.
