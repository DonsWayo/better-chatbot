Phase 2 of the [collaboration phasing plan](/docs/collaboration/teamspaces#electricsql-phasing-plan):
an [ElectricSQL](https://electric-sql.com) sync service streams **reads only** out of
Postgres logical replication. Every write still goes through the existing routes and
server actions — Electric never accepts writes and the app never bypasses its own
authorization, budgets, or audit paths.

## Architecture

<Mermaid
  chart={`flowchart LR
    PG[(Postgres\nwal_level=logical)] -->|logical replication| E[Electric sync service\ninternal only, no auth]
    E -->|HTTP shape log| P["/api/realtime/shape\nauthenticated proxy\nshape whitelist + team ACLs"]
    P -->|long-poll / SSE| B[Browser\n@electric-sql/react useShape]
    B -.->|writes (unchanged)| API[Existing routes & actions] --> PG`}
/>

- **Electric sync service** (`electricsql/electric`, dev compose service `electric` on
  host port `3010`; in Kubernetes an optional internal-only Deployment, `electric.enabled`
  in the Helm chart). It connects to the same database as the app (`wal_level=logical`
  required; the dev compose sets the flag on the Postgres service) and serves the HTTP
  shape API.
- **Authenticated shape proxy** — `GET /api/realtime/shape`
  (`src/app/api/realtime/shape/route.ts`). Electric's shape API is unauthenticated by
  design, so it is **never** exposed to browsers. The proxy implements Electric's
  [proxy auth pattern](https://electric-sql.com/docs/guides/auth): it validates the
  session, decides the shape definition server-side, forwards only the Electric
  protocol params (`offset`, `handle`, `live`, `cursor`, `live_sse`), and streams the
  response through with the `electric-*` headers intact (responses are always marked
  `cache-control: private`).
- **Client** — `@electric-sql/client` + `@electric-sql/react` (`useShape`) subscribe
  through the proxy. Shared helpers live in `src/lib/realtime/shapes.ts`.

## Shape whitelist

The proxy serves exactly three shapes; anything else is rejected with `403`. Clients can
pick a whitelisted shape but can never widen one — `table`, `where`, `params`, and
`columns` are pinned server-side from the session:

| Shape | Definition (server-side) | Access check |
| --- | --- | --- |
| `table=chat_message&threadId=<uuid>` | `chat_message WHERE thread_id = $1`, columns `id,created_at,metadata` | `canReadThread` — thread owner, or `team`-visible thread in a team folder the caller belongs to |
| `table=agent_session` | `agent_session WHERE user_id = <caller>` | always scoped to the caller's own sessions |
| `table=asafe_presence&contextType=thread\|folder&contextId=<uuid>` | `asafe_presence WHERE context_type = $1 AND context_id = $2`, columns `id,user_id,context_type,context_id,last_seen_at` | same gate as the context itself — `canReadThread` for threads, `canAccessFolder` (owner or team member) for folders |

Unauthenticated requests get `401`; a valid session asking for a thread it cannot read
gets `403` (same gate as the shared thread page itself).

## Live shared-thread view

The read-only shared view (`/shared/[threadId]`) stays a server component — it already
owns the row → `UIMessage` mapping and the `PreviewMessage` rendering pipeline. A small
client island (`src/components/realtime/live-thread-messages.tsx`) subscribes to the
thread's `chat_message` shape (ids + metadata only) and calls `router.refresh()`
(debounced) when the message set changes, so the server re-renders with the full
pipeline. The shape is a *change signal*, not a second rendering path — that keeps SSR,
i18n, and message rendering in one place and avoids parsing the `json[]` parts column
client-side.

## Near-live shared generation (v1)

When the owner of a team-shared thread generates a response, teammates watching the
read-only view see it **grow during generation** instead of appearing only when it
finishes. No new infrastructure — it rides the exact pipeline above:

1. **Throttled partial persistence (owner's server request)** — while the model
   streams, the chat route (`src/app/api/chat/route.ts`, helper in
   `src/app/api/chat/shared-stream-partials.ts`) upserts the in-progress assistant
   message — same message id the final persist will use, accumulated text so far, and a
   `{ streaming: true, streamingAt }` metadata flag — at most **every ~2.5 seconds**.
   The thread's sharedness is resolved lazily on the first streamed token
   (`isThreadShared`): non-shared and temporary threads get **zero** extra writes.
   Partial writes are fire-and-forget; a failure can never break the owner's stream.
   The user's question is persisted just before the first partial so viewers see it and
   `created_at` ordering stays correct.
2. **Change signal** — each partial upsert bumps `metadata.streamingAt`, which is part
   of the pinned `chat_message` shape columns, so Electric emits an update and the live
   island refreshes the server component. The page re-renders the partially-persisted
   message through the normal pipeline; messages whose streaming flag is fresh show a
   calm "generating…" pulse. Flags orphaned by a crashed stream age out client-side
   after two minutes — no cleanup job.
3. **Final persist (unchanged)** — the existing `onFinish` write overwrites the same
   row with the complete parts and metadata, clearing the flag.

**Privacy is unchanged**: partials are ordinary `chat_message` rows on a thread that is
already team-shared, served through the same shape proxy and the same `canReadThread`
ACL as the finished message. Nobody sees anything they could not read two seconds later
anyway. The owner's own streaming UI is untouched — partial persistence is purely
server-side.

**Why not token-level fan-out (v2)?** True per-token mirroring would need a stream
broker — something to fan one model stream out to N viewers across pods (Redis
pub/sub, a dedicated SSE relay, or Electric write-path support). This v1 deliberately
rides the existing Postgres → Electric read path on EKS instead: ~2.5 s granularity for
zero new moving parts, no new auth surface, and no second source of truth.

## Presence

Phase 3 adds "who else is here" — a calm avatar stack on shared threads and open team
folders showing teammates active in the same context. Presence keeps the same
architecture as everything else on this page: **writes through a Server Action, reads
through the shape proxy**. No websocket side-channel, no extra infrastructure — just a
tiny table (`asafe_presence`) flowing through Electric like any other shape.

### How it works

1. **Heartbeat (write path)** — while a shared thread or team folder is open *and the
   tab is visible*, the client island (`src/components/realtime/presence-avatars.tsx`)
   calls the `heartbeatPresenceAction` Server Action
   (`src/lib/realtime/presence-actions.ts`) on mount and every **30 seconds**. The
   action validates the session, re-checks that the caller may read that context
   (`canReadThread` / `canAccessFolder`), and performs a single idempotent upsert into
   `asafe_presence` (unique on `user_id + context_type + context_id`, bumping
   `last_seen_at`). A hidden tab stops beating — there is no explicit "leave" write.
2. **Shape (read path)** — everyone in the context subscribes to the `asafe_presence`
   shape through the authenticated proxy (see whitelist above). The shape carries ids
   and timestamps only.
3. **Active window** — a user renders as present while their `last_seen_at` is within
   the last **90 seconds** (three heartbeats of slack), excluding the viewer
   themselves. Stale rows simply age out of the window client-side; no cleanup job is
   required for correctness.
4. **Names and avatars** — resolved lazily via `GET /api/realtime/presence-users?ids=…`
   (authenticated; returns `id`, `name`, `image` only — the same exposure level as
   existing member lists).

Presence only mounts on contexts that are actually shared: a `team`-visible thread
inside a team folder (owner's chat view and the read-only `/shared` view) or an open
team folder in the sidebar. Private threads and personal folders never heartbeat and
never show avatars.

### Typing indicators

Typing rides the same presence row — no extra table, no extra shape. While someone is
composing in a shared thread, their heartbeat carries `typing = true` (sent at most
once every 4 seconds); 5 seconds of silence — or sending the message, or leaving —
clears it. Readers show a muted *"Name is typing…"* line next to the avatar stack for
rows whose typing flag is fresh (last seen within ~10 seconds). Only the boolean flag
travels — never the draft text.

### Privacy

Presence is intentionally coarse: it records *"this user had this shared thread/folder
open within the last 90 seconds"* — nothing else. No cursor positions, no typing
content (the typing indicator is a boolean, not keystrokes), no read receipts per
message. The shape never exposes emails or any user
fields beyond the id; heartbeats stop the moment the tab is hidden or the context is
closed; and the data only flows to people who could already see the same context (the
shape proxy applies the identical access checks as the content itself).

## Phasing

- **Phase 2 (shipped)** — read-path-only sync: live shared threads and the
  caller-scoped `agent_session` shape (proxied, ready to consume). A push-based
  Runs rail over that shape is planned; today the rail and `/runs/[id]` page use
  short-interval polling while a run is live — see
  [Runs & Inbox](/docs/agent-platform/runs-and-triage#live-updates).
- **Phase 3 (shipped)** — presence heartbeats + avatar stacks over the
  `asafe_presence` shape (table-backed, so it rides the existing Electric pipeline).
- **Phase 4 (shipped, first slice)** — typing indicators on the presence row.
- **Phase 5 (shipped, v1)** — near-live shared generation via throttled partial
  persistence (see above); token-level fan-out would need a stream broker and stays
  on the v2 list.
- **Next** — shape subscriptions for folder trees and unread badges, and write
  attribution for multiple humans in one thread.

## Configuration

| Env var | Where | Meaning |
| --- | --- | --- |
| `ELECTRIC_URL` | app | Base URL of the Electric sync service the proxy forwards to. Default `http://localhost:3010` (dev compose). On Kubernetes the Helm chart injects the internal Service URL automatically when `electric.enabled=true`. |
| `DATABASE_URL` | electric | Postgres connection string (user needs the `REPLICATION` attribute). In Helm it is read from the existing app secret (`electric.databaseUrlSecretKey`, default `POSTGRES_URL`). |

Dev bring-up: `docker compose -f docker/compose.yml up -d postgres electric` (the
Postgres service runs with `wal_level=logical`; changing that flag recreates the
container but keeps the data volume), then `curl http://localhost:3010/v1/health` →
`{"status":"active"}`.
