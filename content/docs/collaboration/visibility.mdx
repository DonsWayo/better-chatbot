> "Workflows can be private, shared, team or company — think in all things like that."

Every shareable entity on the platform converges on **one four-level visibility model**
(`docs/design/visibility-model.md`):

| Level | Who can see/use | Who can grant |
|---|---|---|
| `private` | Owner only — **the default for everything**. | — |
| `shared` | Explicitly named users (grant list). | Owner. |
| `team` | Members of one or more teams (`teamIds[]`). | Owner if member; admins. |
| `company` | The whole org. | **Admins only** — the curation gate. |

## Three orthogonal axes

Kept separate on purpose:

1. **Visibility** (above) — who can find/see it.
2. **Capability** — view / use / edit / manage: an entity can be usable org-wide but
   editable only by its owner + admins.
3. **Lifecycle** — draft / published:
   [revisions](/docs/agent-platform/revisions-and-publishing) gate what the org-wide
   audience actually *runs*.

And entitlements always apply **on top**: visibility never bypasses
[model allow-lists](/docs/governance/model-entitlements), tool gates,
[budgets](/docs/governance/budgets-and-usage), or
[guardrails](/docs/governance/guardrails).

## The resolver (implemented)

The shared resolver lives in `src/lib/visibility/` and backs every migrated
entity — no per-feature bespoke checks. Two layers:

- **`resolveAccess(entity, viewer, capability)`** — pure rules, fully unit-tested:
  1. The **owner** and **org admins** always hold `manage` (and therefore everything).
  2. `company` → everyone may `view`/`use`; `edit`/`manage` stay owner + admins.
  3. `team` → members of **any** of the entity's `teamIds[]` may `view`/`use`.
  4. `shared` → the viewer's best grant decides; a grant of a higher capability
     implies every lower one (`manage` ⊃ `edit` ⊃ `use` ⊃ `view`).
  5. `private` → nobody else, ever. Grants are **only** consulted at the
     `shared` level.
- **`canAccess(entityType, entityId, userId, capability)`** — DB-backed gate that
  loads the entity, the caller's admin flag, team memberships and `entity_grant`
  rows, then delegates to `resolveAccess`. Unknown entities and not-yet-wired
  entity types fail **closed**.

**Legacy mapping happens at the resolver, not in the data**: old
`public` values resolve as `company`; old `readonly` resolves as `company`
capped at `view` for non-owners; old `private` (or anything unknown) resolves
as `private`. No rows are rewritten.

**Grants** are stored in the generic `entity_grant` table
(`entityType`, `entityId`, `granteeUserId`, `capability`, `grantedBy`), unique per
(entity, grantee, capability) — granting twice is an idempotent no-op. Sharing is
managed through the `grantAccessAction` / `revokeAccessAction` / `listGrantsAction`
server actions, each of which requires the caller to hold `manage` on the entity
(owner or org admin).

## Implementation rules

1. **One shape everywhere**: `visibility: "private"|"shared"|"team"|"company"`,
   `teamIds: uuid[] | null`, plus the generic `entity_grant` table
   (entityType, entityId, granteeUserId, capability) for `shared` — **implemented**
   (migration `0036_entity_grants`).
2. **One resolver**: `canAccess(entity, userId, capability)` in `src/lib/visibility/`
   — **implemented**, mirroring how
   [model policy](/docs/governance/model-entitlements) and
   [autonomy](/docs/agent-platform/approvals-and-autonomy) resolve org → team → user. No
   per-feature bespoke checks once an entity migrates onto it.
3. **`company` always passes the admin curation gate** — the same gate as featured MCP
   servers and org-wide revisions today.
4. **One `VisibilityPicker` component** (lock / link-user / users / building icons),
   reused everywhere, with the multi-team combobox from the MCP catalog —
   **implemented** (`src/components/visibility/`) and wired into the **workflow
   editor** and **agent editor**. Selecting **team** reveals the team
   multi-select (seeded from the viewer's memberships via
   `/api/teamspaces/folders`); selecting **shared** reveals an inline grant
   manager (add by email → `resolveGranteeByEmailAction`, capability select,
   remove) backed by the `grant`/`revoke`/`listGrants` actions; **company** is
   disabled for non-admins behind a tooltip (the curation gate).

## UI ↔ legacy-column mapping (current caveat)

The workflow and agent rows still store the **legacy** `visibility` enum
(`public | private | readonly`) — widening that column to the literal
four-level value is a future migration this round intentionally does **not**
make. So the editor maps the picker's four-level value onto the legacy column
on save (`toLegacyVisibilityColumn` in `src/components/visibility/`):

- **`company` → writes `public`** (the only value the resolver's legacy path
  reads as org-wide).
- **`private` / `team` / `shared` → write `private`.** The *real* access signal
  for `team` and `shared` lives in the `teamIds[]` column and the
  `entity_grant` table, which the resolver already prefers over the legacy
  enum. So a `team`-shared workflow is stored as legacy `private` **plus**
  `teamIds`, and `canAccess` still grants team members `use`.

Seeding the picker is the inverse (`fromLegacyVisibilityColumn`): `public` →
`company`; legacy `private` with a non-empty `teamIds` → `team`; otherwise
`private`. (`shared` is recognized by the presence of grants, not the row, so
the picker simply opens at the inferred base level and the grant list does the
rest.) A future migration that widens the enum to store the literal value will
make this mapping a no-op.
