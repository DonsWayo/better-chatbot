Per the platform blueprint, asafe-ai does **not** build code/PR agents — engineering uses
its native tooling. The one exception is the desktop: it embeds the open-source
[opencode](https://opencode.ai) server (`@opencode-ai/sdk`) as a **governed coding
surface**. Engineers get a local coding agent on their own machine, but its **model calls
route through the asafe OpenRouter gateway** — so coding sessions inherit the same
[model entitlements](/docs/governance/model-entitlements),
[budgets](/docs/governance/budgets-and-usage), and
[audit trail](/docs/governance/audit-and-compliance) as chat. The desktop never holds
provider API keys.

> **Status:** lifecycle management is implemented (`desktop/src/opencode-manager.ts`) and
> **disabled by default** (ADR-0010). There is no chat/coding UI yet. The web-side
> gateway **is implemented** (`src/app/api/gateway/openrouter/`) — endpoints below.

## How it works

```
Electron main process (opencode-manager.ts)
  └─ gate check: ASAFE_DESKTOP_OPENCODE=1  AND  policy.allowSpawn   ← default deny
  └─ locates `opencode` (ASAFE_OPENCODE_BIN, else PATH) — "unavailable" if absent
  └─ spawns `opencode serve --port 0 --hostname 127.0.0.1`
       env OPENCODE_CONFIG_CONTENT = scoped provider config (never touches user config):
         provider.asafe → @ai-sdk/openai-compatible
           baseURL = ${ASAFE_APP_URL}/api/gateway/openrouter
           apiKey  = {env:ASAFE_SESSION_TOKEN}
  └─ parses the chosen port from stdout → endpoint http://127.0.0.1:<port>
  └─ crash restart cap (3), SIGTERM on app quit (SIGKILL after 5s)
```

The renderer controls it via `window.asafeDesktop.opencode.status() / .start() / .stop()`.
Lifecycle states: `stopped` → `starting` → `running`, plus `unavailable` (gate closed or
binary missing — a helpful message, never a throw) and `error` (spawn failure or restart
cap exhausted).

## The gateway (web side — implemented)

The gateway lives at `src/app/api/gateway/openrouter/` and exposes two endpoints under
`${ASAFE_APP_URL}/api/gateway/openrouter`:

| Endpoint | Purpose |
|---|---|
| `POST /api/gateway/openrouter/chat/completions` | OpenAI-compatible chat completions, streaming (`stream: true`, SSE pass-through) and non-streaming. Matches opencode's `@ai-sdk/openai-compatible` provider plugin. |
| `GET /api/gateway/openrouter/models` | The caller's **entitled** model list — `{ object: "list", data: [{ id, object: "model", name }] }` — so the desktop can populate opencode's model map. |

**Auth (both endpoints):** `Authorization: Bearer <ASAFE_SESSION_TOKEN>`, where the token
is the **raw value of the better-auth session cookie** (`better-auth.session_token`, or
`__Secure-better-auth.session_token` on https deployments) — the signed
`<token>.<signature>` string exactly as stored in the browser. The gateway validates it
against better-auth server-side; provider API keys, OAuth access tokens, or bare session
ids are rejected. The desktop never holds provider credentials — the gateway injects the
server-held `OPENROUTER_API_KEY` upstream and never echoes it in errors.

**Models:** the request's `model` may be the friendly id (`gpt-5.1`) or its OpenRouter
slug alias (`openai/gpt-5.1`). It must be on the approved short list **and** in the
caller's effective allow-list per the [layered entitlements](/docs/governance/model-entitlements)
(org base + team override + per-user grants).

**Budgets/audit:** `checkBudget` runs before forwarding; usage (prompt/completion tokens
from `response.usage`, or the final SSE usage chunk when streaming — the gateway forces
`stream_options.include_usage` upstream) is recorded against the same org/team budgets as
chat (ADR-0003). Each completion writes a `gateway_completion` audit event with
`{ model, originSurface: "opencode", stream }` — never any prompt content.

**Error codes** (JSON body `{ "error": { "message", "code" } }`):

| Status | `code` | Meaning |
|---|---|---|
| 400 | `invalid_request` | Body is not a JSON object. |
| 401 | `unauthorized` | Missing/invalid bearer session token. |
| 402 | `budget_exhausted` | The team's budget cap is reached. |
| 403 | `model_not_allowed` | Model not on the approved list, or not entitled for this user/team. |
| 429 / 4xx / 5xx | `upstream_error` | Relayed from OpenRouter (status passthrough, message sanitized). |
| 502 | `upstream_unreachable` | OpenRouter could not be reached. |
| 503 | `gateway_not_configured` | `OPENROUTER_API_KEY` is not set on the server. |
| 504 | `upstream_timeout` | OpenRouter did not return response headers within 60s (established streams may run longer). |

## Security posture

- **Disabled by default** — without `ASAFE_DESKTOP_OPENCODE=1` the manager is inert:
  nothing is probed, spawned, or written.
- The env opt-in is **interim**: a signed, server-distributed policy plane replaces it
  before GA (`defineOpencodePolicy()` is the seam). The env var then becomes a local
  opt-in ANDed with the signed policy — never a bypass.
- The opencode server binds **loopback only** (`127.0.0.1`, OS-assigned port).
- Governance config is injected via `OPENCODE_CONFIG_CONTENT` (highest-precedence runtime
  override) — the user's global/project opencode config is never modified.

## Enabling it (dev only)

```bash
# In desktop/:
ASAFE_DESKTOP_OPENCODE=1 \
ASAFE_OPENCODE_BIN=/usr/local/bin/opencode \
ASAFE_APP_URL=http://localhost:3000 \
pnpm dev
```
