Model access resolves through **three layers, most specific wins** (ADR-0009), in
`src/lib/admin/model-policy.ts` and `src/lib/admin/user-grants.ts`:

## Layer 1 — org base

A single org-wide allow-list stored under the app-settings key
`org_base_model_allow_list` (`getOrgBaseModelAllowList` / `setOrgBaseModelAllowList`).
`null` means **unrestricted** — every registered model is allowed.

The registered short list is eight models (live OpenRouter pricing on this
account, June 2026). The four **routing-tier** models are what "Auto" selects
between — the entire cost stack lands at $0.10–$3.41 per million tokens (`hy3-preview`, $0.06/$0.21, stays registered for entitled picks but is too slow — 12–19s reasoning latency — for the snappy tier):

| Tier | Model | $/M in | $/M out | Context |
|---|---|---|---|---|
| frontier | `kimi-k2.6` | $0.68 | $3.41 | 262k |
| balanced | `deepseek-v4-pro` | $0.43 | $0.87 | 1M |
| fast | `deepseek-v4-flash` | $0.10 | $0.20 | 1M |
| cheap | `deepseek-v4-flash` | $0.10 | $0.20 | 1M |

The **premium** models — `gpt-5.5` ($5/$30), `claude-opus-4.8` ($5/$25),
`gemini-3.5-flash` ($1.50/$9), and `gemini-3.1-flash-lite` ($0.25/$1.50) —
remain registered but are **entitlement-only**: Auto never routes to them, so
they only run when a team policy or user grant explicitly allows them.

> **Note — MiniMax M3 removed.** `minimax-m3` was the intended frontier-tier
> model, but this account's OpenRouter data policy blocks every endpoint that
> serves it (404 "no endpoints matching your guardrail restrictions and data
> policy"), which surfaced as user-facing errors. `kimi-k2.6` stands in at the
> frontier tier. Re-enabling MiniMax M3 requires loosening the account's
> OpenRouter privacy settings — a compliance decision, not a code change.

## Layer 2 — team override

Each team can carry a `modelPolicy` in one of two modes (`resolveModelAllowList`):

| Mode | Behavior |
|---|---|
| `replace` | The team's list **replaces** the org base entirely. |
| `inherit` | Start from the org base, then apply `add: []` and `remove: []` deltas. |

A legacy per-team `modelAllowList` column is still honored — it is treated as
`{ mode: "replace" }` when no structured policy exists. `resolveTeamModelAllowList(teamId)`
returns the effective team list; unknown teams fall back to the org base, and **DB errors
fail open per layer** (an outage never locks every model).

## Layer 3 — user grants

`asafe_user_model_grant` rows give individual users **additive** access to specific
models, with optional expiry (`expiresAt`) and provenance (`grantedBy`). Expired grants
are excluded at read time; results are cached for 30 seconds and invalidated on write
(`getUserModelGrants`).

The same org → team → user layering is reused for
[autonomy caps](/docs/agent-platform/approvals-and-autonomy#one-resolver-the-entitlements-pattern) and is the
designated pattern for future tool-level entitlements — **one resolver, no parallel
permission systems**.

## Where it's enforced

- The **model picker** only offers entitled models; "Auto"
  ([routing](/docs/architecture#request-path-chat)) only selects among them.
- The **chat route** re-validates server-side — picking a model is a UI affordance, not
  the access check. At request time the three layers are resolved into one effective
  allow-list (`resolveEffectiveModelAllowList`: org base → team policy → user grants as
  additive overrides). That list is passed into the router, so **Auto can never route to
  a non-entitled model**; an explicit pick outside the list returns a 403 naming the
  model. An empty/unset resolution means unrestricted.
- The **desktop opencode gateway** resolves models through the same layers and rejects
  non-entitled models — see [Governed coding](/docs/desktop/governed-coding).
- Visibility never bypasses entitlements: a `company`-visible workflow still runs under
  the *invoking user's* model allow-list (see [Visibility](/docs/collaboration/visibility)).

## Admin surface

Admins manage the org base list, per-team policies, and user grants from the admin area
(`/admin`), including expiring trial grants ("give this analyst the frontier model for two
weeks"). All changes land in the [audit log](/docs/governance/audit-and-compliance).
