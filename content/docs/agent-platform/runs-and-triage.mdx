## The Runs rail (ephemeral)

The Runs rail in the chat sidebar is the platform's **live ticker**: it renders **only
while at least one session is non-terminal** (queued / running / awaiting approval /
paused). The teal pulse dot â€” the one "[alive](/docs/design-language)" signal â€” marks
running sessions, with cost-so-far and abort affordances.

When nothing is live the rail disappears entirely. Completed-run history lives in the
[Inbox](#the-inbox-inbox) under its **Runs** tab.

### Live updates

The rail and the `/runs/[id]` page refresh automatically while a run is in flight: the
rail revalidates `GET /api/runs` every 5s and the per-run page every 4s, **only** while a
session is non-terminal (idle = no requests). Inserts, status changes
(`queued â†’ running â†’ awaiting_approval â†’ completed`) and cost bumps appear without a
manual refresh.

A push-based rail over the `agent_session`
[Electric shape](/docs/collaboration/realtime) (proxied server-side, scoped to
`WHERE user_id = <caller>`) is planned. It was reverted from this build because mounting
the shape long-poll in the always-present sidebar keeps a connection open on every page,
so pages never reach network-idle; the lazy / network-idle-safe wiring is tracked as a
follow-up.

Data still flows over routes for the fallback and the per-id read (per the
[Server Actions vs routes rule](/docs/development/server-actions-vs-routes)):

| Route | Shape |
|---|---|
| `GET /api/runs` | The user's sessions (default page of 30). Degraded fallback for the rail; also `?origin=schedule` filters by `originSurface` (Inbox's "Recent routine runs"). |
| `GET /api/runs/[id]` | One session with its ordered step rows (`getSessionWithSteps`). |
| `GET /api/agent-platform/approvals/count` | Pending-approval count â€” the badge on the sidebar Inbox item. |

Mutations â€” abort, steer â€” are Server Actions in `src/app/api/runs/actions.ts`. An abort
shows instantly via an optimistic overlay, then the authoritative status arrives over the
shape.

## Agent chats are governed sessions

A chat turn run **with a selected agent** (an `@agent` mention) is now a first-class
governed run, exactly like a workflow run. On each turn's `onFinish`, the chat route
records a `conversational` `agent_session` for the thread (one per thread, reused across
turns, keyed on `input_payload.threadId`) and appends the userâ†’assistant exchange as a
`turn` [agent_step](/docs/agent-platform/sessions-and-steps), with the turn's cost rolled
into `cost_so_far` from the same usage it meters for billing. The agent's published
[revision](/docs/agent-platform/sessions-and-steps#agent_session--one-governed-execution)
is pinned when one exists.

So an agent chat shows up in the Runs surfaces with a per-turn transcript + cost, just
like a workflow run. Recording is **fully non-blocking and fail-open** (a persistence
error never breaks the chat â€” `recordAgentChatTurn` in
`src/lib/agent-platform/conversational-session.ts` swallows). Stored turn input/output is
a **bounded, content-safe preview** (trimmed to 2000 chars + a hash + char count â€” the
same "no raw bodies" discipline the
[audit log](/docs/governance/audit-and-compliance) follows), never a raw message dump.

**Plain (no-agent) chats are intentionally NOT governed**, to keep run volume bounded â€”
only agent chats earn a session.

## /runs/[id] â€” permanent transcripts

Every session has a permanent transcript page at `/runs/[id]`
(`src/app/(chat)/runs/[id]`), built from the
[checkpointed step rows](/docs/agent-platform/sessions-and-steps#agent_step--resumable-checkpoints):
node-by-node inputs/outputs, per-step cost, timings, and failures.

The blueprint's audit story goes one step further: the run URL is injected into
tool-execution context so agent outputs in Jira/Confluence/email can say *"produced by
asafe-ai run \<link\>"* â€” the linkback compliance signs off on.

## The Inbox (/inbox)

`/inbox` (`src/app/(chat)/inbox`, formerly `/triage` â€” a server redirect protects old
deep links) is the one always-visible sidebar destination for things that need a human.
It carries a pending-approvals badge and is rendered for **everyone**, even with zero
runs. Tabs:

- **Inbox** â€” approval requests the current user can decide
  (`listPendingApprovalsForUser` respects the
  [`requestedRole`](/docs/agent-platform/approvals-and-autonomy#the-approval-gate) rules), with
  approve/reject and a reason field, plus recent routine runs
  (`originSurface: "schedule"`), so routine findings land somewhere visible instead of
  an unread dashboard. The design goal is signal-only: empty runs auto-archive.
- **Runs** â€” the full recent-run history (all kinds, with status and cost), replacing
  the old always-on rail as the home of completed runs.
- **Routines** â€” the user's scheduled workflows (enable/disable, next/last run).

## Visibility of runs

Runs follow their container (see the [visibility model](/docs/collaboration/visibility)):
the initiator and admins always see a run; sessions inside a shared team folder become
readable to folder members (phase 1 is read-only) â€” steer/abort stays restricted to the
initiator and folder admins.
