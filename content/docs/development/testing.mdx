Every wave shipped with tests across three tiers:

| Tier | Command | Cost | When |
| --- | --- | --- | --- |
| Unit (Vitest) | `pnpm test` | free | every change, CI |
| E2E (Playwright) | `pnpm test:e2e` | free | integrated flows, CI |
| Real-LLM | `RUN_LLM_TESTS=1 pnpm test:llm` | ~$0.01/run | opt-in, never in CI by default |

The unit suite currently counts **5,500+ test cases**
(`*.test.ts` colocated next to their modules) plus Playwright e2e under `tests/`.

## Unit tests (Vitest)

```bash
pnpm test          # run once
pnpm test:watch    # watch mode
```

Conventions:

- **Colocated** — `worker.ts` is tested by `worker.test.ts` in the same directory.
- **No live database.** Library code is tested against mocked Drizzle surfaces, not a
  Postgres instance (e2e covers the integrated path).

### The db-mock pattern

Server lib modules import three things that need stubbing in tests — the pattern (from
`src/lib/agent-platform/sessions.test.ts` and hundreds of siblings):

```ts
// 1. Mock the db handle: chainable query-builder stubs
vi.mock("lib/db/pg/db.pg", () => ({ pgDb: { /* select/insert/update chains */ } }));

// 2. Mock the schema module: table objects can be plain placeholders
vi.mock("lib/db/pg/schema.pg", () => ({ AgentSessionTable: {}, /* … */ }));

// 3. Mock drizzle-orm operators used by the module under test
vi.mock("drizzle-orm", () => ({ eq: vi.fn(), and: vi.fn(), /* … */ }));

// 4. server-only would throw outside Next.js
vi.mock("server-only", () => ({}));
```

Step 4 is also handled globally: `vitest.config.ts` aliases `server-only` to the no-op
stub in `src/lib/__mocks__/server-only.ts`, and `vite-tsconfig-paths` resolves the repo's
path aliases inside tests.

Pure modules (the [cron engine](/docs/agent-platform/routines#the-cron-engine), guardrail
patterns, the entitlement resolvers) are deliberately dependency-free so they test
without any mocking — prefer that shape when adding logic.

## Real-LLM integration tier

Unit tests mock every model call. The opt-in real-LLM tier (`*.llm.test.ts`,
run by `vitest.llm.config.ts`) makes **real OpenRouter calls** to catch what
mocks structurally cannot — stale model slugs, prompt regressions, structured
output (`generateObject`) robustness on cheap models:

```bash
RUN_LLM_TESTS=1 pnpm test:llm   # needs OPENROUTER_API_KEY in .env
```

Rules of the tier:

- **Opt-in twice.** Every file guards with
  `describe.skipIf(!(process.env.OPENROUTER_API_KEY && process.env.RUN_LLM_TESTS === "1"))`
  — without both, the suite **skips** (never fails). `pnpm test` excludes
  `**/*.llm.test.ts` entirely.
- **Costs real money** — roughly **$0.01 per run**. Assertions use only the
  cheapest approved models (`gemini-3.1-flash-lite`, `deepseek-v4-flash`) with
  tiny output caps; the registry smoke pings every approved model with
  `max_tokens: 1`.
- **Never in CI by default.** Run it locally after touching `src/lib/ai/models.ts`,
  routing policy, extraction/title prompts, or workflow generation.
- Serial by config (`fileParallelism: false`, 30s timeouts); `.env` is loaded
  via `src/lib/load-env.ts` as the setup file.

Current coverage: the model registry smoke (`models.llm.test.ts` — every
approved slug must answer HTTP 200, so a stale slug can never ship silently),
routing → real completion (`route-model.llm.test.ts`), memory extraction
roundtrip (`extract.llm.test.ts`, store mocked), NL workflow generation against
the real tool schema (`generate-workflow.llm.test.ts`), and the refusal-proofed
thread-title prompt (`prompts.llm.test.ts`).

## E2E (Playwright)

```bash
pnpm test:e2e:seed   # seed test users (dotenv + scripts/seed-test-users.ts)
pnpm test:e2e        # run; :ui for headed mode
pnpm test:e2e:clean  # remove all test data
```

The auth proxy returns `pong` on `/ping` so Playwright's webServer readiness check works
unauthenticated.

## The gate

`pnpm check` = Biome lint:fix + `tsc --noEmit` + Vitest. The definition of done for every
change also includes: env vars documented in `.env.example`, schema changes with
[committed migrations](/docs/development/migrations), and metrics on new request paths.
